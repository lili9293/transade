How to create an .xml file while using the TRANSADé schema
This section provides schemata to help:
create .xml files based on the TRANSADé schema (also known as (transade).xml files)
and also read the reports create by the application.

The (transade).xml file
This is the main file of the application. One can compare it to the default "index.html" file in a web application or the main class in a java application.
The root always has to be the transade. Which in turn is made of two principal nodes: the references - and the transfer's.

The (references).xml file
As its name stipulates the references is used as references repository. It occurs at most once in a (transade).xml file. To avoid useless repetitions this node enable the usage of certain specific blocks more than once without having to write the corresponding statements more than once.
There are two ways to successfully write a references.:
It can be directly written in the (transade).xml file (check the (transade).xml file - box to see how it is done).
It is also possible the write an extra file uniquely made of the references (hence the name (references).xml file) and to include it in the its respective (transade).xml file. Check the (references).xml file - box to see how it is done.
The advantages of the options are the better overview and the possibility of using one references for more than one (transade).xml file.

The (transfer).xml file
The transfer is the second child of the transade. It should contain the logic that eventually will determine how to transfer the data. It can be be created just like references's. Yet unlike them the transfer occurs at least once.
Each transfer is made of one or more source's, which in turn contain one or more target's. The combination of one source and its target's is defined as one computation unit. Each source or target is defined a specific format and its respective definitions. Click here formats & definitions for more information.
In order to enable flexibility and performance the following processes have been established.:
Inside of a specific transfer the computation units are handled sequentially. Meaning the first source will be the first to be parsed, compiled and executed. The second node will the second and so on.
However all selected transfer's will be started and processed concurrently.

The (report transade).xml file
Once the computation of all the selected transfer's done, the application generates a report file named report.pdf. This file gives an overview of how good the parsing stage was and how successful the compiling and execution stages were.

How to use the TRANSADé graphic user interface
This chapter presents the most useful and basic features of the gui. Just like on the GUI the progress (top to bottom) in this chapter event based.

The configuration
Right after running the application -for the first or without having ever configure it the following warning will be prompted.



The settings section is located at the top right of the GUI and is represented by the symbol. Once selected, the settings/configuration frame menu will pop up.



Once the working directory and the language set, the application will be good to go. This process can be repeated indefinitely.

The files upload
The application provides two ways to upload (transade).xml files.
The most common way is to use the file chooser on the top left side of the GUI.



It's important to note that only -xml files are accepted by the file chooser.
The second process is to use to choose one the previously saved (transade).xml file templates. They are listed in the drop down list right next to the file chooser stated above.
No further actions are required for the download. Once done the application presents all the transfer's in form of a list.

The nodes selection
Once a (transade).xml file is successfully uploaded all the transfer's are presented in form of a checkbox list of their identifier in the current page. The identifier is the value attribute id (click here the (transade).xml file for more information). To choose which transfer's to execute simply check its checkbox.



At times before choosing certain transfer's one might first need to take a look at logic residing in the given node. To avoid going back and fort between the application GUI and the .xml editor the GUI provides a handy feature allowing the user to have a selective look at the original file. By right clicking the transfer identifier a tree representing the node will appear in the section located in the east canter of the GUI. This tree enables a view of the node. Note: a node doesn't have to be selected to be viewable.



There are 10 pages in total that can used simultaneous to handle 10 different files, each of them is equipped with a lamp system.



There are 3 lamps in total, each of them representing one possible page status.
The (also known as the default lamp) means that the page has never been used before. As one can imagine it is the default lamp/status.
The (also known as the good to go lamp) indicates that a file as been successfully downloaded on the page and everything is good to go.
The (also known as the end lamp) signalizes that a file as been successfully downloaded; some or all transfer's residing in within have been parsed, compiled and eventually executed.

The nodes execution
The node execution section resides at the upper south side of the GUI. The execution is pretty straight forward. Once all the transfer's selected, simply set the 3 running options and click the symbol.



The options are from the left to the right.:
The computation mode. 3 modes are available.
PREVIEW: simply parse the transfer's and create the adequate .scala classes.
It's recommended to use this mode in the early stage of the development.
BETA COMPILE: parse the transfer's and create, compile and execute the adequate .scala classes.
It's recommended to use this mode once the result of the PREVIEW mode are satisfying, before the development, to compare the results and the .scala classes.
USER COMPILE: parse the transfer's and create, compile, execute and delete the adequate .scala classes.
This is the mode to used during the development.
The decision about whether or not the report file (report.pdf) should be opened at the end of the computation.
(optional) And the possibility of saving the loaded files, the selected transfer's and the setting of the options stated above as one template.
